📐 Design Annotations — Click to expand UX/UI explanation of each section
▲
01 — Sidebar
Global Triage Panel
The Today's Triage 2×2 grid replaces the old sidebar's generic "Active: 9" count with semantically colored numbers (red = overdue, amber = due tomorrow, green = active deals, neutral = closing soon). This answers "what needs my attention right now" without leaving the sidebar.
Navigation is restructured around workflow stages — Active Deals, Deadlines, Documents, Contacts, Messages — not developer data types. Each nav item with live data gets a color-coded badge (red for overdue, amber for due soon, green for unread messages).
The "+ New Transaction" CTA is pinned to the sidebar footer — always reachable, never competing with content. Dark sidebar creates a natural visual hierarchy separating navigation chrome from work surface.
02 — Header
Persistent Status Ribbon
The persistent status ribbon (4 overdue · 5 due tomorrow · 3 closing · 2 unread) is always visible regardless of scroll position. This replaces the current design's "0 tasks / 0 overdue" badge bar that disappears when scrolling down to transactions. A TC should never have to scroll up to see their status summary.
The greeting is condensed to a single line with the date and deal count inline — useful context, not decorative space consumption. In the original Screen 1, the greeting header consumed 12% of vertical space.
AI Credits are surfaced as a contextual pill next to the search bar — not buried in a corner. This positions AI as a workflow accelerator, not a hidden setting.
03 — Pipeline Strip
Above-The-Fold Intelligence
4 pipeline summary cards are the first content element above the fold — immediately answering the agent's morning question: "What's happening in my pipeline today?" Each card has a color-coded top accent bar using the semantic color system (red = closing soon, amber = in inspection, green = active, blue = pending).
Each card shows not just a count but the single most relevant contextual detail (e.g. "Next: Mar 6 — Connors", "2 responses overdue"). This is dashboard intelligence — the interface anticipates the follow-up question and answers it preemptively.
In the original Screen 1, "Upcoming Closings" was below the fold — the most commission-critical section of the dashboard buried beneath empty task cards. This redesign promotes it above the transaction list.
04 — Transaction Cards
Collapsible Card System
Each card header shows exactly 6 data points at-a-glance: client name, address, stage pill, milestone timeline, next critical date, and task/doc badges. This replaces Screen 2's 4-column horizontal layout that required reading across 1200+ pixels per transaction.
The left-edge colored indicator bar (red/amber/green) allows instant triage scanning down a list of 9+ deals — the eye goes straight to red bars without reading any text. This is a pattern borrowed from calendar apps and email clients where users have trained their peripheral vision to spot urgency.
The visual milestone timeline replaces the separate Inspection Timers / Closing / Possession columns from Screen 2. The timeline communicates stage progression at a glance — filled dots = completed, pulsing dot = current stage, empty dots = upcoming, red dot = overdue. This preserves the semantic distinction between inspection and closing dates while reducing visual noise by 60%.
Expanded card body is a 3-column grid: Tasks (inline checkboxes for completion without navigation) · Key Dates (all critical dates with status coloring) · Contacts (one-click call/email directly from the card). This brings all one-click-away actions to the surface, eliminating the navigation drilling that the audit identified.
The AI Assist button appears contextually in the card footer — "Summarize this deal", "Draft inspection response", "Closing checklist" — surfacing AI as a deal-specific tool, not a generic feature.
05 — Right Column
Closing Countdown + Global Tasks
The Closing Countdown widget uses visual countdown tiles (4 days = red, 17 days = amber, 26 days = green) so a TC can orient to timeline urgency instantly. This replaces Screen 2's approach of showing closing dates buried inside individual transaction cards. Closings are surfaced as their own first-class element.
The "Needs Attention Today" widget is a global cross-deal task view — something missing entirely from both original screens. A TC managing 9 deals shouldn't have to open each transaction card to build their daily to-do list. Tasks are linked back to their deal with a green deal name tap that navigates to that transaction.
The 340px right column sits at a natural reading width that doesn't compete with the transaction cards on the left. At common desktop resolutions (1440px+) both columns are fully visible without scrolling for the most critical 2-3 deals.
06 — Color System
Semantic Token Architecture
A unified semantic color system is applied consistently across every element: Red (#e53b3b) = overdue/critical. Amber (#d97706) = due soon/warning. Green (#1a9e72) = completed/healthy/on-track. Blue (#3b82f6) = informational/pending. Gray = inactive/empty. This replaces the inconsistent orange/gold (Screen 2) vs teal (Screen 1) conflict that created confusion about what urgency signals meant.
Every color is implemented as a CSS variable — meaning a design system change to one token propagates everywhere automatically. This is production-ready architecture, not just a mockup aesthetic choice.
Status pills use a background-tint + border + text approach rather than solid fills, maintaining readability while allowing the left edge indicator bar to carry the full urgency weight for rapid visual scanning.
07 — Filter Bar
Persistent Filtering & Sort
A persistent filter chip bar (All / Overdue / Closing Soon / In Inspection / Pending) allows a TC to instantly narrow a 9+ deal list to a specific workflow stage without navigating away. This replaces the complete absence of filtering in both original screens.
The "Overdue (4)" chip badge is always visible, creating urgency even when the user is in a neutral viewing state. A TC who has been in a document review for 20 minutes and returns to the dashboard sees immediately that 4 tasks are overdue — they don't have to scroll to discover this.
Sort defaults to "Urgency" — most overdue and soonest-closing transactions appear at the top. This matches the mental model of every working TC: triage by risk, not alphabetically or by date created.
08 — What Was Preserved
Honoring What Worked
The "New Transaction" button placement in the sidebar is preserved exactly — it was correctly positioned in Screen 1 and is a reliable pattern for professional SaaS apps.
The triage count mental model from Screen 2's left sidebar (Overdue: 4, Due Today: 0, Due Tomorrow: 5) is preserved and elevated to the sidebar's permanent header section, now visible across every page not just the transactions view.
The semantic distinction between inspection dates and closing dates from Screen 2 is preserved in the expanded card's Key Dates section — these are legally distinct events that TCs track separately, and the redesign maintains that logic.
The unread messages badge tied to individual transactions (not a global inbox) is preserved architecturally — communication is scoped to a deal. The badge is now hidden when count = 0, appearing only when there are unread messages.
