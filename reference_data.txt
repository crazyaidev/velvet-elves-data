Project Overview
Velvet Elves is a web‑based transaction management platform designed for real estate brokerages. The current version, implemented in ASP.NET MVC/C#, orchestrates every stage of a residential real‑estate transaction: from creating a deal record, assigning tasks and managing vendors, to storing and emailing documents and communicating with clients. Core functions include document storage, wizard‑generated task lists based on different use cases, a Gmail‑based email API, user accounts, payment processing for transaction fees/commissions, and dashboards tailored for agents, team leaders, transaction managers, clients and administrators. Users log in via account invitations, register or reset passwords, and then work on transactions through a responsive web interface. The application heavily relies on task dependencies, document workflows, contact management and automated email notifications.
This scope document outlines the functional modules, desired improvements and new AI‑driven features to guide a developer in estimating the cost and timeline of rebuilding Velvet Elves. Where specific details are unknown, reasonable assumptions are noted; please adjust those assumptions as needed.
Core Modules & Required Features
1. User Management & Authentication
Account registration and login – The existing system sends invitation tokens and allows recipients to register or log in through password‑reset workflows. Maintain secure registration, login, password reset and invite‑based onboarding. Consider upgrading to modern authentication (e.g., OAuth2/JWT) for security and future scalability.
Role‑based access control – Support the following roles: Agent, Team Leader, Transaction Manager, Client (buyer/seller), and Administrator. Each role has distinct permissions for viewing, editing and approving transactions and tasks. Administrators and team leaders can manage other users, assign transactions and configure the system. Vendors (e.g., mortgage/title) may be represented as linked contacts rather than full users, but the design should allow additional roles to be defined or adjusted via configuration.
Contact management – The current system stores details for co‑agents, loan officers, title reps, buyers and sellers, and provides functions to update names, emails and phone numbers. Maintain a centralized contact directory linked to transactions and vendors.
2. Transaction Management
Transaction lifecycle – Users can create, edit and delete transactions. The model includes address fields, representation type (buyer, seller or both), purchase price, financing type (cash or financed), closing date and notes. The system loads incomplete transactions and populates vendor lists, state lists and agent lists. Reimplement this with intuitive forms and built‑in validation.
Task automation based on wizard inputs – When a transaction is created or edited, the system automatically assigns “generic” tasks based on conditions (e.g., financing type, title order, representation). These tasks include acceptance and closing tasks, financing‑specific tasks, and contingency tasks. The new implementation should preserve this rule engine and make it configurable so that brokers can adjust which tasks are triggered by which conditions.
Transaction assignment – Transactions can be assigned to an elf or an agent. Assignment triggers notifications and additional task creation. Support reassignment and allow multiple participants per transaction (agents, elves, vendors, buyers, sellers).
Filtering & reporting – The system currently returns filtered lists of transactions for different roles (elf, agent, vendor, client) with sorting, searching, and pagination. Recreate this functionality with modern data‑table UX (filter by status, closing date, representative, location, etc.) and export capabilities (CSV/Excel/PDF).
Payment processing – Integrate payment workflows so agents can collect transaction fees and commissions. The existing codebase references a separate payment module. The rebuild should support secure credit‑card processing, invoicing, tracking payments and refunds, and integration with accounting platforms. Payment actions must trigger tasks or status updates as appropriate.
Dashboards for each role – Provide tailored dashboards for agents, team leaders, transaction managers, vendors, clients, and administrators. These dashboards should summarize active transactions, upcoming deadlines, outstanding payments, and communication status at a glance. Allow customization of widgets and include interactive charts or KPIs.
3. Task Management
Task creation & editing – Users can add, edit and delete tasks associated with transactions and the master task lists for each use case. Tasks have due dates, description, dependencies, owners and optional “float” (buffer days). The system also supports tasks that depend on wizard questions or other tasks. Rebuild this module to allow: creation of templates, assignment of tasks to users, and visual management of dependencies (e.g., Gantt or dependency tree).
Dependency engine – Tasks can depend on other tasks or conditions; they may not start until predecessors complete. Provide a rule builder so administrators can define dependencies without code changes.
Test task management – The system includes a “Test Task” section allowing creation of generic tasks not tied to any transaction. Decide whether to retain this feature; if retained, move it into a template/task library module.
Notification & reminders – Tasks trigger email reminders to assignees when due or critical. Implement configurable reminder intervals (e.g., due today, past due), push, and in‑app notifications.
Reporting – Provide dashboards to show tasks by status (previous, today, upcoming) and highlight tasks with upcoming deadlines. Send a daily summary email when there are tasks that are due and not when there aren’t.
Comprehensive task catalogue – The legacy REWORKING TASK DB.xlsx file contains a structured catalogue of more than fifty tasks that guide a transaction from contract acceptance through closing and post-closing follow-up. These tasks include, among others:
Welcome & communication tasks – “Buyer Welcome,” “Seller Welcome,” “Co‑op Agent Welcome,” “Loan Officer Welcome,” and “New Transaction Added” notifications, many of which are automated and triggered on contract acceptance.
Documentation & compliance tasks – “Review Documentation,” “Request HOA Docs,” “Deliver HOA Docs,” “Order Title,” “Confirm Title Order,” “Buyer Documentation,” “Deliver Title,” “Deliver Utility Info,” “Title Work Completed,” and “Closing Disclosure Delivered.”
Vendor & contingency tasks – “Order Home Warranty,” “Confirm Home Warranty Order,” “Insurance Reminder,” “Inspection Scheduled,” “Inspection Completed,” “Inspection Response Reminder,” “Inspection Negotiated,” “Appraisal Ordered,” “Appraisal Completed,” and “Clear to Close.”
Closing & follow‑up tasks – “Schedule Closing,” “Buyer Closing Information,” “Seller Closing Information,” “Buyer's Agent Closing Information,” “Seller's Agent Closing Information,” “Closing Gift,” “Request Referrals,” “Exemptions and Thank You,” “Thank You,” and “Internal Thank You.”
Meta tasks – “Closing Date,” representing the key date used as a reference for tasks with floats.
 Each task entry specifies its maturity relative to its dependency (e.g., on/Before/After another task or specific date), the float or offset in days, and the applicable cases (Buy-Fin, Buy-Cash, Sell-Fin, Sell-Cash). Tasks marked as “Automated” or “To Be Automated” indicate opportunities for system automation and AI integration. The rebuilt system should import this catalogue as the default task library, allow administrators to edit task definitions (name, dependencies, float, case applicability, automation flag), and add new tasks as needed.
4. Document Management
Upload & storage – Agents, elves, and clients can upload documents (e.g., purchase contracts, disclosures). Store files securely in a document center linked to transactions.
Document retrieval & metadata – The system provides a “GetDocumentList” API returning document name, type, and ID; users can rename or delete documents. Maintain metadata and versioning so users can track changes and view previous versions.
Document emailing – The existing controller can send one or more documents via email to selected recipients (transaction participants, clients) and logs each email as a transaction note. Rebuild this with improved UI: choose recipients, subject, body and attachments; use templated emails for consistency.
AI‑assisted document processing – New feature: integrate AI to read uploaded documents (contracts, PDFs) and auto-extract key fields (buyer names, closing dates) into structured data for easier entry. This will reduce manual data entry and improve accuracy.
5. Communication & Notifications
Email notifications – The system sends emails when transactions are created, tasks are assigned or updated, and documents are shared. Use a robust email service with templating and logging. Support CC/BCC and attachments.
Client onboarding – Clients receive registration or login emails with unique tokens when they are added to a transaction. Maintain this workflow and consider adding two‑factor authentication for improved security.
Communication log – Maintain a log of all communications per transaction (emails sent, phone calls, notes) for auditability.
AI‑powered email replies – New feature: integrate generative AI to answer routine client questions and requests via email. For example, a client can email the system asking for a document or status update, and the AI assistant can respond automatically by retrieving the requested information from the transaction records and attaching the correct document. Include fallback to human review for complex inquiries.
6. Vendor & Integrations
Vendor management – The system maintains lists of mortgage and title vendors, along with preferred vendors for transactions. Allow administrators to add, edit and deactivate vendors.
Advertising & sponsorship – Provide the ability to sell advertising space within the wizard and task modules to local vendors (e.g., title companies, mortgage lenders, inspectors, contractors). This requires an ad management interface where administrators can create and price advertising packages, approve vendor creative assets (logos, banners, contextual offers) and schedule where and when ads appear (for example, during task creation for inspections or loan contingency tasks). Track impressions and clicks to report performance to advertisers. Ads should be clearly delineated from system content and must not compromise user experience or security. Support different ad formats (display banners, text links, sponsored suggestions) and ensure they are responsive across devices. A billing component should tie into the payment module to invoice and collect fees from advertisers.
External integrations – Integrate with third‑party services that are essential to a real‑estate transaction workflow:
MLS feeds – Pull property and listing data directly from the Multiple Listing Service to pre‑populate transaction details.
E‑signature providers (e.g., DocuSign, HelloSign) – Obtain digital signatures for contracts and disclosures.
Customer Relationship Management (CRM) systems – Sync leads, contacts and communications with the brokerage’s CRM. Allow two‑way data flow so new clients in Velvet Elves appear in CRM and vice versa.
Title company software – Exchange files and status updates with title agencies to streamline escrow and closing. Support common file formats and secure messaging.
Mortgage Loan Officer (MLO) software – Connect with MLO platforms to share loan status, requirements and documents, reducing manual updates.
Calendar & email APIs – Leverage Gmail API (existing) and calendar APIs (Google Calendar, Outlook) to schedule appointments, send reminders and manage events.
Payment module – The original repository includes a separate module (VEPaymentModule). If integrated, specify requirements for invoicing, commission tracking or integration with accounting software.
7. AI & Automation
AI‑assisted coding – The development team is expected to use AI coding agents (e.g., Copilot, ChatGPT) to accelerate implementation. Provide guidelines on code quality, documentation and human oversight.
Natural‑language task creation – Allow users to create tasks by describing them in plain language; an AI agent converts the description into structured task objects. Example: “Send inspection contingency reminder three days before inspection” turns into a task with proper due date and reminder rules.
Predictive analytics – In later phases, consider using AI to predict closing delays, identify transactions at risk, or prioritize tasks based on historical data.
8. User Interface & Experience
Responsive web application – The primary platform should be a responsive web app accessible from desktops and mobile browsers. Use a modern front‑end framework (e.g., React, Angular) and ensure that key dashboards (transactions, tasks, documents) are intuitive and load quickly.
Mobile support – In addition to the responsive web app, plan for a supporting native or hybrid mobile app in a subsequent phase (assuming the timeline and budget permit). The mobile app should enable agents to view and update transactions, upload documents via camera and receive push notifications.
Dashboard & reporting – Provide a customizable dashboard summarizing open transactions, upcoming closings, overdue tasks, and new documents. Include filters and quick links to jump into specific transactions or tasks.
Accessibility & internationalization – Ensure the app meets basic accessibility guidelines (WCAG AA) and allows for multilingual support if your brokerage serves multilingual clients.
White‑label customization – Allow brokerages or partner companies to purchase and deploy the platform under their own branding (“white‑boxing”). This includes the ability to upload custom logos, set brand colors and themes, and optionally configure a custom domain or subdomain. Each tenant’s branding should be applied across the login screens, dashboards, email templates, and documents without impacting the core functionality. Consider a multi-tenant architecture to isolate data and branding per customer.
9. Data Migration & Security
Data migration – Only the task lists from the legacy system will be migrated into the new platform. Because documents, transactions, and user accounts will be recreated or imported manually, the migration scripts should focus on exporting existing task templates and assignments from SQL Server, transforming them to match the new schema, and loading them into the fresh database. Provide tools or guidance for brokers to re‑enter or import transactions and contacts as needed. Ensure that any essential historical data not being migrated is archived and accessible offline.
Security – Protect personally identifiable information (PII) through encryption at rest and in transit. Enforce least‑privilege access for all roles. Comply with relevant regulations such as CCPA/GDPR. Implement audit logging for critical actions.
Backup & disaster recovery – Outline backup strategies and disaster recovery plans to ensure business continuity.

Velvet LMS is a mostly automated transaction management platform for residential real estate transactions.

VENDOR COMMUNICATION FEATURE. “LINK RESPONSE” VS “AI READS EMAIL”
Old system: you emailed vendors with a link they clicked (“done/not done”), and if not done they provided a date, which updated the task maturity date.
New option you proposed: have AI read the vendor’s email reply and update the task date automatically if it doesn’t violate overall timeline constraints.
Suggested approach:
•	Yes, let AI parse the email, but keep it structured.
 Best practice is a hybrid:
1.	Email asks for a date in a constrained format (“Reply with: Scheduled: YYYY-MM-DD”).
2.	AI parses reply and proposes the updated task date.
3.	If the reply is vague (“soon,” “next week,” “working on it”), the AI either:
	Sends a clarification email automatically, or
	Routes to a human, depending on confidence.
DOUBLE-CHECK MECHANISM
•	Read the message/doc.
•	Produce an interpretation.
•	Re-read and compare against its first interpretation.
•	Proceed only if both agree with high confidence; otherwise ask for clarification or human review.
•	Make the second pass use a different method (different prompt strategy, different extraction template) so it’s not just repeating the same mistake confidently.
•	Require agreement on key fields only (date, party, signature status, requested deliverable), not on every word.
 
 
STATE-BASED DIFFERENCES - FOR A MASTER TASK LIST
You asked about “title vs deed states,” and then pivoted into “title theory vs lien theory vs intermediate theory.” Important note: those are different concepts, and mixing them will create bad automation rules.
What you actually need for transaction processing is usually:
A) Closing/settlement model by state (attorney closing vs title/escrow closing)
This affects:
•	Who prepares settlement docs.
•	Who conducts closing.
•	Where money sits (escrow), how disbursement works, and timing.
•	Which party manages payoff, recording, and post-close curatives.
B) Mortgage/foreclosure theory (lien vs title vs intermediate)
Actionable guidance for Velvet Elves master tasks:
•	Build state rules around “closing practice” (attorney vs title/escrow), “recording timelines,” and “disbursement timing,” not just lien/title theory labels.
•	Your workflow differences from pending-to-close should key off:
1.	Who runs closing (attorney vs title/escrow company).
2.	Whether there’s required attorney involvement in doc prep/settlement.
3.	Recording and funding sequence expectations (some places record and disburse quickly, others have timing quirks).
4.	Local customary forms/requirements and compliance steps (state disclosures, local addenda norms, etc.).
