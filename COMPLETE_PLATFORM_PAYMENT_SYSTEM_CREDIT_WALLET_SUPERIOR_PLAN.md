# Complete Platform Payment System Establishment Plan (Credit-Wallet, Superior)

Created: 2026-06-18
Author: Jan (sole developer)
Status: plan only. No source code is changed by this document.

Purpose: stand up a complete, end-to-end platform payment system that charges real-estate professionals to USE Velvet Elves, runs flawlessly in Stripe test mode against a provisional base price, and lets Jake and non-developer testers validate every step from the frontend UI before launch. The first objective is not to finalize commercial pricing. The first objective is to prove the mechanism works and feels like a professional tool.

This plan supersedes the subscription-oriented `COMPLETE_PLATFORM_PAYMENT_SYSTEM_ESTABLISHMENT_PLAN.md` (written by OpenAI Codex). Section 2 records exactly what I keep from that plan and what I change, and why. The decisive change, confirmed by Jan: bill per transaction through a non-expiring credit wallet, not a recurring SaaS subscription.

## 0. Source-verification review and corrections (2026-06-18)

I re-read the live source against the first draft of this plan to find workflow and logic errors before any code is written. This section is authoritative: where the body below (Sections 1 onward) conflicts with a correction here, the correction wins. Each item names the file I verified so the plan cannot drift from the code. I have also edited the affected body sections to match.

### R1 (critical). The spend cannot be debited "before" create; the transaction id does not exist yet

The first draft (Section 6.4) said to call `try_spend(tenant_id, prospective_transaction_id, ...)` "as the first step" and to make it "idempotent on transaction_id". That is impossible: `TransactionRepository.create` generates the id itself with `tx_id = str(uuid.uuid4())` (`app/repositories/transaction_repository.py:83`), so the API layer has no transaction id until after the row is inserted. There is also a concurrency race if I only read the balance before creating.

Correction, mirroring the existing seat pattern (`SeatService` does an app-side pre-check while the DB function `create_invitation_with_seat_check` is the authoritative atomic gate):

1. App-side fast pre-check: read the wallet balance. If it is 0, return `402` and create nothing. This handles the common empty-wallet case without ever creating a row.
2. Create the transaction (the id is generated by `repo.create`).
3. Authoritative atomic debit: call a Postgres RPC `spend_credit_for_transaction(p_tenant_id, p_transaction_id)` that, in one DB transaction, locks the wallet row, re-checks `balance >= 1`, decrements, and inserts the `credit_ledger` spend row (with `transaction_id` and `balance_after`). `balance_after` is computed inside the RPC, never in app code, so it is correct under concurrency.
4. Compensation: if the RPC fails because a concurrent create consumed the last credit (pre-check passed, authoritative debit lost the race), delete the just-created transaction and return `402`. The row is milliseconds old with no documents yet, so the delete is clean.

Idempotency, restated correctly: the unique partial index on `credit_ledger (tenant_id, transaction_id) WHERE entry_type='spend'` guarantees at most one recorded debit per transaction. `create_transaction` itself is not idempotent across network retries today (a retry creates a second transaction), and that is unchanged by billing: two distinct transactions correctly cost two credits. Double-submit is the same pre-existing UX concern (disable the button while the mutation is pending), not a new billing bug.

### R2. Reversal-on-delete is Admin/Team Lead only, not owner, and must read the ledger before delete

The first draft (Section 5.5) said the in-window reversal is for "Admin / TeamLead / owner". The delete endpoint is `require_role(UserRole.ADMIN, UserRole.TEAM_LEAD)` (`app/api/v1/transactions.py:790`); a plain owner whose role is Agent cannot delete a transaction. Correction: the reversal runs inside `delete_transaction` and is therefore Admin/Team Lead only. Also, the transaction FK uses `ON DELETE SET NULL` (the endpoint's own docstring confirms this for documents), so the spend ledger row's `transaction_id` becomes NULL once the row is deleted. The reversal must therefore look up the spend row by `transaction_id` and credit it back BEFORE `repo.delete(tx)` runs, and must be a no-op if the spend was already reversed.

### R3. Grant free credits on first wallet creation, not on tenant creation

The first draft (Section 5.3) hooked the free-credit grant to tenant creation. There are at least four tenant-insert paths (`app/repositories/tenant_repository.py` lines 54, 87, 191, 223: registration, backfill, and others), so a tenant-creation hook is fragile and easy to miss. Correction: grant the free credits lazily inside `credit_wallet_service.get_or_create_wallet`, guarded by a `free_credits_granted` flag on the wallet, so the grant happens exactly once on first use regardless of how the tenant was created, and shows up as a visible `grant` ledger row.

### R4. A credit-purchase refund needs its own admin endpoint; the existing refund path will reject it

The first draft (Section 6.3 and tester script 8) implied an admin could refund a credit purchase, but the existing `RefundService.create_refund` loads a `payments` row and raises `PaymentNotFoundError` when there is none (`app/services/refund_service.py:59-63`). Credit purchases never create a `payments` row, so that endpoint cannot refund them, and "use the Stripe dashboard" violates the no-developer-tools rule. Correction: add a platform-admin endpoint `POST /api/v1/platform/billing/purchases/{id}/refund` that calls `stripe_client.create_refund(payment_intent_id=...)` using the purchase's stored payment intent, and extend `_sync_refund_state` in the dispatcher to record the reversal for a credit purchase (right after the existing ad-order fallthrough, since neither has a `payments` row). Tester script 8 then stays fully in the UI.

### R5. The 402 paywall rides the existing create hook and the existing draft/resume machinery

Verified and strengthened. The wizard's create call is the typed mutation `useCreateTransaction` (`src/hooks/useWizardApi.ts:304`), declared `useMutation<Transaction, ApiResponseError, ...>`, and `apiFetch` throws `ApiResponseError(status, body)` for any non-2xx while only invoking the auth-failure handler on 401 (`src/utils/api.ts`). So a `402` is catchable at the hook with its body intact. The wizard also already autosaves a server-side draft (`wizard_runs`: `get_current` / `upsert_current` / `delete_current`, one draft per user) and already shows a "Resume draft" prompt. Correction/refinement: do not invent a new resume route. Persist `return_context` on the `credit_purchases` row so the success URL is `/transactions/new` (which auto-offers the existing draft resume) for a mid-deal purchase, and `/organization?section=billing` for a Billing-page purchase.

### R6. Active-workspace billing is automatic; no special billing scope code is needed

Verified. `apiFetch` always sends `X-Workspace-Id` from `getActiveWorkspaceId()` (`src/utils/api.ts`), and `get_current_user` re-scopes `user.tenant_id` to the active workspace when the header is present and the flag is on (`app/core/auth.py:106-119`). Because the wallet keys on `current_user.tenant_id`, billing already follows the active workspace and a guest already spends the host workspace wallet, with no extra billing-layer logic. This is a simplification, not new work.

### R7. The frontend auth fields exist but are optional; treat undefined as false

Verified. `is_tenant_owner?` and `is_platform_admin?` exist on the user type (`src/types/api.ts:26,32`) and `hasMinimumRole(userRole, requiredRole)` exists (`src/utils/constants.ts:16`). Both flags are optional booleans. Correction: `canManageBilling = Boolean(user?.is_tenant_owner) || hasMinimumRole(user.role, 'Admin') || Boolean(user?.is_platform_admin)`.

### R8. The single create choke-point is confirmed

`create_transaction` (`app/api/v1/transactions.py:171`) is the only POST that creates a transaction; `preview-tasks` and `{id}/tasks/generate` operate on an already-created transaction. So gating the debit at `create_transaction` on the backend covers the AI wizard, the intake flow, and quick-create alike. The friendly paywall only needs to be surfaced wherever `useCreateTransaction` is consumed; any path I miss still fails safe with a backend 402 rather than an uncharged transaction.

## 1. Why this plan exists, and the failure it fixes

Earlier plans broke during frontend testing because they were drafted without a full reading of the requirements, system design, and the live frontend/backend source. The workflow looked right on paper and then fell apart in the UI, which is fatal because the testers are real-estate professionals, not developers. They cannot pause webhooks, run the Stripe CLI, read a database row, or recover a half-built flow.

I wrote this plan only after reading the live source. Every architectural claim below names the file it is grounded in, so the plan cannot drift from the code. The build target is a payment system where a tester, using only a mouse and the Velvet Elves frontend, can: buy credits, watch the balance rise, create a deal, watch the balance fall, run out, hit a calm paywall, buy more, and keep working. No terminal. No Stripe dashboard. No dead-end pages.

## 2. Decision summary and relationship to the Codex plan

### 2.1 The model: per-transaction credit wallet

Velvet Elves charges for usage through a prepaid, non-expiring credit wallet, in the style of ListedKit (the competitor Velvet Elves is modeled on). One credit is spent when a user creates a transaction. Credits are purchased in packs through the existing Stripe Checkout rails. Team members are free; there is no per-seat fee in this system.

Why this model, grounded in source:

- The existing Stripe wrapper only supports `mode='payment'` checkout (`velvet-elves-backend/app/services/stripe_client.py`, `create_checkout_session`). Credit packs are one-time payments, so they reuse it as-is. A subscription model would require net-new `mode='subscription'` methods, Customer Portal, and recurring-invoice webhooks that do not exist today.
- The webhook receiver and idempotency gate already exist (`app/api/v1/payments.py`, `POST /webhooks/stripe`, unique `stripe_event_id` in `webhook_events`). Credit grants ride this unchanged.
- The dispatcher already branches `checkout.session.completed` by `metadata.kind` (`_is_ad_order` runs before the invoice path in `app/services/payment_event_dispatcher.py`). A `kind=credit_purchase` branch slots in at the same seam.
- There is exactly one place a transaction is created: `create_transaction` at `app/api/v1/transactions.py:171`. A single, well-understood spend choke-point is far less likely to break in testing than entitlement checks scattered across the app.
- "Set a base price first, decide the exact price later" maps to one configurable number (price per credit / per pack). Jake can change it without a code change.
- The Organization page already exposes a deep-linkable section rail and a read-only "Plan" placeholder whose help text literally says "Online plan changes arrive once pricing is finalized" (`velvet-elves-frontend/src/pages/organization/OrganizationPage.tsx:356`). The wallet UI fills that exact slot.

### 2.2 What I keep from the Codex plan

The Codex plan is a careful document and several of its principles are correct regardless of model. I keep:

- Three separate money lanes that must never share entitlement logic, reporting labels, or accounting meaning: platform billing (workspace pays Velvet Elves), tenant workflow payments (workspace invoices clients), advertising (advertisers buy placements). See Codex Section 1; I preserve it in Section 4.5 here.
- Honest test-mode language: never label a provisional amount as the final price; always show a test-mode badge where checkout can start; never hardcode a price in a component. Codex Section 4.1.
- Hosted Stripe only. No card data touches Velvet Elves. PCI SAQ-A. Codex Section 4.3, confirmed by the PCI note in `stripe_client.py`.
- Billing authority anchored on the owner, not the legacy edit check: `canManageBilling = is_tenant_owner || Admin || platform admin with audit context`. Codex Section 2.1.
- No dead-end return pages. Stripe returns into a real app surface, not an isolated route. Codex Section 8.2.
- A local/dev-only test harness so non-developers can exercise delayed-webhook and replay states without the Stripe CLI. Codex Section 6.2 / 8.6.
- Active-workspace scoping: billing follows the active workspace and honors `X-Workspace-Id` when multi-workspace is enabled. Codex Section 2.1.
- Platform-admin price console so Jake can review and change the base price from the UI. Codex Section 8.3.

### 2.3 What I change from the Codex plan, and why

| Codex plan | Change in this plan | Reason (source-grounded) |
| --- | --- | --- |
| `mode='subscription'` checkout, plans + seats. | `mode='payment'` credit-pack checkout; per-transaction debit. | The wrapper has no subscription methods; credit packs reuse `create_checkout_session` unchanged (`stripe_client.py`). |
| `billing_subscriptions`, `_items`, `_invoices`, `billing_entitlements` mirror tables. | `credit_wallets`, `credit_ledger`, `credit_purchases` only. | No recurring objects to mirror. The ledger is the single source of truth and is trivially auditable. |
| New webhook handlers for `customer.subscription.*`, `invoice.paid`, `invoice.payment_failed`, `invoice.payment_action_required`. | One new branch `kind=credit_purchase` on the existing `checkout.session.completed` handler; reuse `charge.refunded` for reversals. | The configured webhook endpoint already lists exactly these events (`STRIPE_LOCAL_CONFIGURATION.md`); no new Stripe event types are needed. |
| Stripe Customer Portal for card/invoice management. | Not required for v1. Receipts come from Stripe automatically on each one-time payment; purchase history is shown from `credit_purchases`. | Removes a whole net-new Stripe surface and its return-flow failure modes. Can be added later if Jake wants saved cards / auto-refill. |
| Entitlements degrade through `past_due / grace / restricted / suspended`. | No recurring dunning. The only gate is "do you have a credit to spend?" Out of credits is a calm paywall, never a lockout of existing work. | Per-transaction billing has no renewal to fail, so the entire dunning state machine and its hard-to-test states disappear. |
| Seat limits enforced as a paid entitlement. | Members are free; the seat-billing gate is retired behind a flag. | Today nothing in the UI ever sells a seat, so the seat gate only ever blocks testers (`seat_service.py`, `assert_staff_seat_available`). Retiring it removes the most failure-prone user-management branch. The `tenants.plan` / `seat_limit` columns are left in place but dormant. |
| Trial as a checkout or card-on-file decision. | Trial stays a local state; optionally grant N free credits on first wallet use (configurable, default first deal free; see R3). | `tenants.trial_ends_at` exists but is not enforced today; free credits are simpler and fully UI-visible. |

Net effect: the same launch goal (a complete, testable platform payment system on Stripe test mode with a provisional base price) with a fraction of the new Stripe surface, no recurring-billing failure states, and a workflow a non-developer can validate top to bottom.

### 3.3 What is missing for platform billing

- No way for a workspace to pay Velvet Elves for its own usage. All current Stripe flows are client-facing (invoices, refunds, commission transfers, advertising).
- No wallet, no ledger, no concept of a spendable balance.
- No spend gate on transaction creation.
- No platform-admin surface to set or review the base price.

### 3.4 Constraints the new system must respect

- Do not introduce new `tenants.plan` keys; the DB CHECK is `('trial','solo','team','enterprise')` (per `REVENUE_GENERATION_SYSTEM_PLAN.md` correction C4). Credits are orthogonal to plan, so no plan-key change is needed.
- Tenant isolation is strict and service-role based. Wallet/ledger/purchase tables must carry `tenant_id` and be scoped exactly like every other tenant table.
- `delete_transaction` is a hard delete. The ledger's `transaction_id` reference must not block deletion (use `ON DELETE SET NULL`), and a deletion may trigger a credit reversal (Section 6.6).
- Billing reads/writes must resolve through the active workspace, honoring `X-Workspace-Id` where multi-workspace is enabled. A guest spends the host workspace's wallet (resolves the open M3 question in `MULTI_WORKSPACE_MEMBERSHIP_DESIGN_PLAN.md`).
- Tenant workflow invoice payments and advertising payments must keep working untouched. The credit branch runs before the invoice path so a credit purchase never marks an invoice paid and never creates a `payments` row.

## 4. Product principles

### 4.1 Honest, provisional pricing

Final pricing is undecided, so the base price is a configurable value with provisional, test-mode framing:

- Store the per-credit price and the pack ladder in configuration / a small table, never in a component.
- Every place a purchase can start in local/dev shows a "Test mode - provisional pricing" badge.
- Never present a test amount as the final launch price.
- Jake can change the price from the platform-admin Base Price console (Section 8.3) with no code change.

### 4.2 Mouse-first, minimal typing

A real-estate professional should complete every billing action with the mouse. The only typing ever required is card entry inside Stripe's hosted page, which Velvet Elves never sees. Concretely:

- Buy credits: open Billing, click a pack, click "Buy", pay on Stripe, return. Done.
- See the balance, see the history, see what a deal costs: all read-only, always visible.
- Run out mid-deal: a single "Buy credits" button on the paywall, then the deal resumes by itself.

### 4.3 Hosted Stripe only

Card collection is Stripe Checkout (`mode='payment'`). Velvet Elves stores only safe summaries (amount, pack name, last4 if Stripe returns it). PCI scope stays SAQ-A.

### 4.4 Never strand active work

Out of credits blocks one thing only: starting a new transaction. It never blocks viewing, editing, or progressing existing deals, documents, tasks, or communication. The paywall is calm and recoverable, not a lockout.

### 4.5 Three money lanes stay separate

- "Credits / Billing" = the workspace paying Velvet Elves to create deals.
- "Invoices and Payments" = the workspace collecting money from clients and transaction parties.
- "Advertising" = advertisers buying public placements.

These share Stripe infrastructure and the one webhook pipeline, but they never share entitlement logic, ledgers, or reporting labels.

## 5. The money model: per-transaction credit wallet

### 5.1 Unit and meaning

- The spendable unit is a credit (integer). One credit is consumed when a transaction is created.
- Credits never expire.
- The whole team shares one wallet at the tenant/workspace level. Members are free.

### 5.2 The base price (one configurable number)

- A single per-credit price drives everything (placeholder for testing, for example $X.00 per credit).
- Packs are quantity bundles of credits at that price, with an optional bulk discount. Placeholder ladder: 1, 5, 10, 25 credits.
- All values come from configuration / a small `credit_packs` source, surfaced to Jake in the Base Price console. The exact numbers are placeholders until Jake decides; the code reads the config, not a literal.

### 5.3 Free credits (trial replacement)

- Grant N free credits lazily on first wallet creation (configurable, default: first deal free, matching ListedKit's "first one free"), guarded by a `free_credits_granted` flag so it happens exactly once regardless of which tenant-creation path ran (per R3). This is a ledger `grant` row, fully visible in the history, not a hidden flag.

### 5.4 The spend event (corrected per R1)

- One credit is consumed per transaction, debited around `create_transaction` (`transactions.py:171`) in the order set by R1: app-side balance pre-check, then create, then an atomic `spend_credit_for_transaction(tenant_id, transaction_id)` RPC that decrements and writes the ledger row together, then compensation (delete the new row) on the rare lost-race.
- If the pre-check finds balance 0, `create_transaction` returns `402 Payment Required` with `{ reason: "insufficient_credits", balance: 0, buy_options: [...] }` and creates nothing.
- The unique partial index on `credit_ledger` guarantees one recorded spend per transaction. Two distinct transactions correctly cost two credits; `create_transaction` is not idempotent across retries today and billing does not change that.
- Alternative spend point (open question O3 for Jake): debit when a deal first becomes Active rather than at create. Default is debit-at-create because it is the single endpoint and the most directly testable; the alternative is recorded for Jake.

### 5.5 Refund / reversal of a spend (corrected per R2)

- `delete_transaction` is a hard delete (`repo.delete` runs `.delete().eq("id", ...)`), and it is restricted to Admin and Team Lead (`require_role(ADMIN, TEAM_LEAD)`, `transactions.py:790`). Deleting a freshly created transaction within a short, configurable window credits the wallet back with a ledger `reversal` row.
- Because the ledger's `transaction_id` is `ON DELETE SET NULL`, the reversal must read the spend row by `transaction_id` and credit it back BEFORE `repo.delete(tx)` runs, and must be a no-op if it was already reversed. The ledger keeps full history; nothing is erased.

### 5.6 Grant on payment, never on redirect

- A credit purchase grants credits only when the Stripe webhook `checkout.session.completed` with `kind=credit_purchase` is processed. The success-return page never grants; it polls and reflects state. This is the same trust boundary the invoice and ad-order flows already use.

## 6. Backend architecture (additive, reuse-first)

### 6.1 New tables (migrations, additive only)

| Table | Purpose | Key columns |
| --- | --- | --- |
| `credit_wallets` | One spendable balance per tenant/workspace. | `id`, `tenant_id` (unique), `balance` int >= 0, `free_credits_granted` bool, `created_at`, `updated_at`. |
| `credit_ledger` | Append-only history; the source of truth. | `id`, `tenant_id`, `entry_type` in (`grant`,`purchase`,`spend`,`reversal`,`adjustment`), `delta` int (signed), `balance_after` int, `transaction_id` FK ON DELETE SET NULL, `purchase_id` FK nullable, `actor_user_id` nullable, `reason` text, `created_at`. Unique partial index on (`tenant_id`,`transaction_id`) where `entry_type='spend'` to enforce one debit per transaction. |
| `credit_purchases` | One row per credit-pack checkout attempt and its outcome. | `id`, `tenant_id`, `pack_id`, `credits`, `amount_cents`, `currency`, `status` in (`pending`,`paid`,`failed`,`refunded`), `stripe_checkout_session_id`, `stripe_payment_intent_id`, `checkout_url`, `return_context` in (`billing`,`wizard`) for the success URL (R5), `started_by_user_id`, `created_at`, `updated_at`. |
| `credit_packs` (optional; may start as config) | Sellable bundles and the base price. | `id`, `key`, `display_name`, `credits`, `unit_amount_cents`, `is_active`, `is_test`, `sort_order`, `updated_by_user_id`, `updated_at`. |

The ledger is authoritative; `credit_wallets.balance` is a denormalized cache kept in lockstep inside the same DB transaction as each ledger insert. All four tables carry `tenant_id` and reuse the existing tenant RLS / service-role pattern.

### 6.2 New services

- `credit_wallet_service.py`
  - `get_or_create_wallet(tenant_id)`: lazily creates the wallet and grants the configured free credits once, guarded by `free_credits_granted` (R3).
  - `get_balance(tenant_id)`: the fast pre-check used by the spend gate (R1 step 1).
  - `spend_for_transaction(tenant_id, transaction_id, actor_user_id)`: thin wrapper over the `spend_credit_for_transaction` Postgres RPC that atomically decrements and writes the ledger spend row (R1 step 3); returns success or "insufficient".
  - `reverse_spend(tenant_id, transaction_id, actor_user_id, reason)`: reads the spend row by `transaction_id` and credits it back; no-op if already reversed (R2).
  - `grant(tenant_id, credits, entry_type, reason, actor_user_id)` for free credits and admin grants.
  - `apply_purchase(purchase_id)`: called from the webhook branch; idempotent; inserts a `purchase` ledger row and bumps the balance once.
  - `record_refund(payment_intent_id)`: maps a refunded intent to a `credit_purchases` row, marks it `refunded`, and writes a `reversal` ledger entry (R4).
- `credit_purchase_service.py`
  - `create_checkout(tenant_id, pack_id, started_by_user_id)`: validates the pack, builds the `create_checkout_session` call with `metadata.kind=credit_purchase`, persists a `pending` purchase row, returns the hosted URL. Modeled directly on `advertising_service.py:450`.
  - `get_status(purchase_id)` for return-page polling.
  - `list_purchases(tenant_id)` for history.
- `credit_pack_service.py` (platform-admin): list/create/update/deactivate packs and the per-credit price; validates positive amounts; marks "ready for Jake review".

No change to `stripe_client.py` is required: credit packs use the existing `create_checkout_session` and `create_refund`.

### 6.3 Webhook branch (one new branch, reuse everything else)

In `payment_event_dispatcher.py`, extend `_on_checkout_completed` so that, mirroring `_is_ad_order`, it checks `metadata.get("kind") == "credit_purchase"` before the invoice path and routes to a new `_on_credit_purchase_paid(metadata, obj)` which:

1. Resolves the `credit_purchases` row by `metadata.purchase_id` (and/or the checkout session id).
2. If already `paid`, returns (idempotent).
3. Marks the purchase `paid`, stores the payment intent.
4. Calls `credit_wallet_service.apply_purchase(...)` to add credits and write the ledger row.
5. Optionally writes a system communication-log entry and an accounting event, matching the invoice path's behavior.

For reversals (corrected per R4): the existing `RefundService.create_refund` cannot be reused, because it loads a `payments` row and raises `PaymentNotFoundError` when there is none (`refund_service.py:59-63`), and a credit purchase never creates a `payments` row. So I add a platform-admin endpoint `POST /api/v1/platform/billing/purchases/{id}/refund` that calls `stripe_client.create_refund(payment_intent_id=<purchase.payment_intent>)`, and I extend the dispatcher's `_sync_refund_state`: when the refunded payment intent has no `payments` row and is not an ad order, try the credit-purchase path, mark the purchase `refunded`, and write a `reversal` ledger entry. This is the same "no payments row -> try ad order -> now also try credit purchase" fallthrough the dispatcher already uses for ads, and it keeps the whole refund test in the UI.

Because the receiver already inserts a unique `webhook_events` row and dispatches in the background, replay safety and fast acknowledgement are inherited unchanged.

### 6.4 The spend gate (corrected per R1 and R8)

In `create_transaction` (`transactions.py:171`), behind the feature flag, in this exact order (the transaction id does not exist until `repo.create` runs):

1. Pre-check: `credit_wallet_service.get_balance(current_user.tenant_id)`. If 0, return `402` with the buy-options payload and create nothing.
2. Create the transaction via `repo.create` (id generated server-side) and run the existing assignment/audit logic.
3. Authoritative debit: call the `spend_credit_for_transaction(tenant_id, transaction_id)` RPC, which locks the wallet, re-checks `balance >= 1`, decrements, and inserts the `credit_ledger` spend row in one DB transaction.
4. Compensation: if step 3 reports insufficient balance (a concurrent create won the last credit), delete the just-created transaction and return `402`.

This mirrors the seat model exactly: an app-side pre-check for fast UX plus an authoritative atomic DB function as the real gate, the same split as `SeatService` and `create_invitation_with_seat_check`. `create_transaction` is the only POST that creates a transaction (preview/generate operate on an existing one), so this is the only consumption point in the system, which is precisely why it is testable and hard to break.

### 6.5 API surface

Tenant-facing, prefix `/api/v1/billing/credits`:

| Method | Path | Purpose |
| --- | --- | --- |
| GET | `/wallet` | Current balance, free-credit state, what one deal costs, recent ledger. |
| GET | `/packs` | Active sellable packs and the per-credit base price (test-mode flagged). |
| POST | `/checkout-session` | Start a credit-pack purchase; returns hosted Stripe URL. |
| GET | `/purchases/{id}` | Poll a purchase after return from Stripe. |
| GET | `/ledger` | Paginated credit history. |

Platform-admin, prefix `/api/v1/platform/billing`:

| Method | Path | Purpose |
| --- | --- | --- |
| GET | `/packs` | List packs + base price + active/test state. |
| POST | `/packs` | Create a pack / set the base price (local/dev). |
| PUT | `/packs/{id}` | Edit display/price/active; require a reason on price change. |
| GET | `/health` | Stripe key mode, webhook backlog, pending purchases stuck, packs missing price. |
| GET | `/tenants/{tenant_id}/wallet` | A tenant's balance + ledger for support. |
| POST | `/tenants/{tenant_id}/grant-credits` | Grant/adjust credits with a required reason (audited). |
| POST | `/purchases/{id}/refund` | Refund a credit purchase from the UI; calls `stripe_client.create_refund` with the stored payment intent (R4; the invoice-bound `RefundService` cannot be reused). |
| POST | `/test-harness/{tenant_id}/simulate` | Local/dev only: simulate delayed webhook, replay, reset wallet. Returns 404/403 in production. |

Permissions:

- Buying credits and viewing the wallet: `is_tenant_owner || Admin` by default (open question O5: whether Team Lead can also buy). Agents/TC can see the balance and the paywall but, by default, the paywall's "Buy credits" routes the request to an owner/admin if they lack purchase permission (so a deal is never silently blocked).
- Platform-admin routes sit under the existing `PlatformAdminGuard` and require platform-admin with audit context.
- All routes resolve the active workspace and honor `X-Workspace-Id`.

### 6.6 Idempotency and integrity rules

- One debit per transaction: enforced by the unique partial index on `credit_ledger` and by the `spend_credit_for_transaction` RPC re-checking inside a row lock.
- One grant per purchase: `apply_purchase` is idempotent on `purchase_id` and guarded by the purchase status.
- Balance can never go negative: the spend RPC decrements only when `balance >= 1` under the row lock.
- Webhook replay never double-grants: inherited from `webhook_events` uniqueness plus the per-purchase guard.
- Hard delete never orphans the ledger: `transaction_id` is `ON DELETE SET NULL`; the history entry remains, optionally annotated by the reversal.

When `create_transaction` returns `402` (corrected per R5 and R8):

The wizard catches `err.status === 402` and shows a calm inline paywall card, not an error toast: "You're out of credits. Buy credits to start this deal." Any other consumer of `useCreateTransaction` (intake, quick-create) gets the same safe 402 and can reuse the shared handler.
